# Understanding and Fixing "Sticky Hover" on Mobile Devices

## The Problem: What is "Sticky Hover"?
On desktop devices, the `:hover` CSS pseudo-class triggers when the mouse enters an element and turns off when it leaves. However, mobile devices (touchscreens) don't have a "mouse cursor" that can leave an element. 

When a user taps a button on mobile, most browsers "emulate" a hover state. Because there is no "mouse out" event, the button stays in the hover state (e.g., staying dark or highlighted) until the user taps somewhere else on the screen. This makes the UI feel "stuck" or broken, especially for toggle buttons like your "Notes" icon.

---

## The Solutions We Implemented

### 1. The `hover` Media Query (CSS)
The most modern and effective way to fix this is using the `@media (hover: hover)` interaction media feature.

**What it does:** It tells the browser to only apply the styles inside the block IF the device has a primary input mechanism that can actually hover (like a mouse or trackpad).

```css
/* This will NEVER trigger on a standard smartphone */
@media (hover: hover) {
  .icon-btn:hover {
    background-color: #1b5b7e;
    transform: scale(1.1);
  }
}
```

We also used `@media (pointer: fine)` in some cases, which specifically targets high-precision pointers like mice, further ensuring touchscreens are excluded.

### 2. Preventing "Sticky Focus" (JavaScript/React)
Sometimes the highlight isn't just "hover"; it's "focus." When you click a button, it remains "focused" by the browser until you click something else. 

We used two tricks in React to combat this:
* **`e.currentTarget.blur()`**: This manually tells the browser to "un-focus" the button immediately after the click event is processed.
* **`onMouseDown={(e) => e.preventDefault()}`**: This is a pro-tip! By preventing the default behavior of the `mousedown` event, we stop the browser from ever giving the button "focus" in the first place, while still allowing the `click` event to fire.

### 3. Explicit State Classes vs. Pseudo-classes
Instead of relying on `:active` or `:focus` to show that the notes were open, we used a React state variable (`isEditingNote`) to apply a real CSS class: `.active`.

**Why?** Because we have total control over a class. We can turn it on and off exactly when we want, whereas pseudo-classes like `:hover` are at the mercy of the browser's internal logic.

### 4. Tap Highlight Color
Mobile browsers often add a default translucent gray or blue box when you tap an element. We disabled this to ensure our custom UI was the only thing the user saw:
```css
-webkit-tap-highlight-color: transparent;
```

---

## Summary for the Future
When building responsive buttons:
1. **Always** wrap your `:hover` styles in `@media (hover: hover)`.
2. **Never** use `:hover` for "Toggle" states; use a JavaScript-controlled class like `.is-open` or `.active`.
3. **Use** `preventDefault` on `mousedown` if you want to prevent that persistent "focused" look on buttons after they are clicked.
